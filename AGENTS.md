# AGENTS.md — kusuriya/nix

Multi-host NixOS flake.Three machines: * * framey * * (daily driver laptop), **beast** (desktop/gaming), **pve** (home server).

## Quick navigation

| Path | What |
|------|------|
| `flake.nix` | Entry point. `mkSystem` helper creates each host config. Hosts listed here. |
| `hosts/<name>/` | Per-host config. Each has `default.nix`, `packages.nix`, `hardware-configuration.nix`, `disko.nix`. |
| `modules/core/` | Shared across all hosts (nix, packages, fonts, locale, kusuriya user). |
| `modules/desktop/` | Desktop shared modules (sway, sysctl, btrfs-snapshots, dozer-mounts, virtualization). |
| `modules/kernel/latest/` | Latest kernel pin. |
| `home-manager/` | Home-manager config + dotfiles under `dotfiles/`. |
| `overlays/` | `unstable-packages` overlay. |
| `pkgs/` | Custom packages (rd560 label printer, glowing-bear). |
| `roles/` | Server roles on pve (mediaserver, paperlessngx, lightning, ingress-controller, dakland). |
| `justfile` | Shortcut commands — `just switch beast`, `just check`, `just update`. |

| `.githooks/pre-commit` | Git hook — auto-formats staged Nix files before commit. Install: `git config core.hooksPath .githooks`. |
| `.github/workflows/check.yml` | CI — runs `treefmt` + `deadnix` on every push. Also does a `nix build` of framey. |

## Adding a new host

Use `mkSystem` in `flake.nix`:

```nix
myhost = mkSystem {
hostname = "myhost";
extraModules = [
# Framey: inputs.hardware.nixosModules.framework-13-7040-amd + common-pc-ssd + lanzaboote
# Beast:  inputs.hardware.nixosModules.common-cpu-amd + common-gpu-nvidia + common-pc-ssd
# Add hardware modules matching the machine's hardware here
];
};
```

Each host directory needs:
- `default.nix` — imports + host-specific config (security, networking, services)
- `packages.nix` — GUI apps, dev tools, host-specific packages (core CLI tools go in `modules/core/packages.nix`)
- `hardware-configuration.nix` — generated by `nixos-generate-config`
- `disko.nix` — disk layout (copy subvolume pattern from existing hosts)

Desktop hosts (framey, beast) import: `modules/core` + `modules/kernel/latest` + `modules/desktop/sway` + home-manager. Server hosts (pve) import roles instead.

**Non-desktop hosts** (like pve) don't use `mkSystem` — write a standalone `nixpkgs.lib.nixosSystem` call. Skip home-manager entirely.

## Build commands

```bash
# Build and switch on the target host
sudo nixos-rebuild switch --flake ~/nix#<hostname>

# Build without switching
nixos-rebuild build --flake ~/nix#<hostname>

# Update flake inputs
nix flake update
```

## Conventions — read this before editing

### Design principles

- **Prefer modular, reusable units over bespoke configs.** Shared patterns (kernel hardening, sysctls, btrbk snapshots, NFS mounts, PipeWire tuning, disko layout) are duplicated intentionally — they compose per-host without cross-contamination. If you find yourself writing the same block in a third host, extract it into `modules/`.
- **Each host owns its trade-offs.** Don't pull host-specific config into core modules just to reduce duplication. Core = truly universal. Host = opinionated about that machine.
- **Document removals and experiments.** If something was tried and removed, leave a comment explaining why. Future agents need to know it was considered, not just absent.

### Code style

- **Comment the WHY, not the WHAT.** If you're adding something that might surprise someone, write the reasoning inline. If removing something, say why. Look at existing comments for tone — they explain trade-offs, failed experiments, and decisions.
- **Section headers** in long files use `# ========` bordered blocks (packages.nix) or `# --- ---` (shorter sections). Match whichever is nearby.
- **Multi-line function args** each on their own line, indented. Example from flake.nix:
```nix
mkSystem = { hostname, system ? "x86_64-linux", extraModules ? [ ], homeManagerConfig ? true }:
```
- **Formatting**: `nixpkgs-fmt` is the formatter. `deadnix` is the linter. Both run via `treefmt`. Run `treefmt` before committing Nix files.

### Architecture rules

- **Core modules** go in `modules/core/`. These apply to ALL hosts. If something is host-specific, put it in that host's `default.nix` or `packages.nix`.
- **Packages** are split: `modules/core/packages.nix` for universal CLI tools (git, jq, ripgrep, bat, etc.). Each host's `packages.nix` for GUI apps, dev tooling, or security tools specific to that machine.
- **The `mkSystem` helper** in flake.nix wires up: host config + core modules + home-manager (for desktop hosts). PVE does NOT use `mkSystem` — it's a standalone configuration without home-manager.
- **Host-specific overlays**: each host imports `self.overlays.unstable-packages` directly. Don't add new overlays globally unless they're really universal.
- **Don't add host modules to `modules/core/default.nix`** — that's the shared import, not a dump.
- **nixpkgs channels**:
| Name | Channel | Access |
|------|---------|--------|
| `pkgs` | `nixos-unstable` | default |
| `pkgs-stable` | `nixos-24.11` | via `specialArgs` in `flake.nix` |
| `pkgs.unstable` | `nixpkgs/master` | via `unstable-packages` overlay |
Use `pkgs-stable` for tools that break often on unstable (libvirt, qemu, virt-manager are already pinned). Use `pkgs.unstable` for packages you need the absolute latest of.
- **Kernel**: `modules/kernel/latest` pins `linuxPackages_latest` (latest stable).

### Security patterns to preserve

- **SSH access**: Laptops (framey) restrict SSH to Tailscale interface only. Desktops (beast) may also open it on LAN interfaces — it's a trusted local network. Servers (pve) SSH on LAN via physical NIC. Decide per host based on trust model.
- Kernel hardening sysctls (kptr_restrict, dmesg_restrict, perf_event_paranoid, ptrace_scope, RP filter, no redirects, no source routing, SYN cookies) are duplicated across framey and beast. If adding a new host, include these.
- AppArmor `killUnconfinedConfinables = true` on desktop hosts.
- `audit.enable = true` and `auditd.enable = true` on desktop hosts.
- `boot.kernelParams` includes `audit=1` on all hosts.
- Do NOT disable these unless there's a specific compatibility reason, and document it.

### Filesystem patterns

- **btrfs everywhere** for / and /home (except pve which uses ZFS). Snapshot config is consistent across framey and beast:
- hourly snapshots via btrbk
- retention: 48h 14d 8w 6m
- snapshots in `/.snapshots`
- auto-scrub monthly
- **disko subvolume layout** (framey + beast): `@root` → `/`, `@home` → `/home`, `@nix` → `/nix`, `@log` → `/var/log`, `@snapshots` → `/.snapshots`, `@persist` → `/persist`. All mounted with `compress=zstd,noatime`. New btrfs hosts should copy this layout.
- **Multi-device btrfs caveat**: disko does not support multi-device btrfs natively. Beast has 3 NVMe drives but disko only manages drive 1. Remaining drives are added to the pool post-install via `btrfs device add`. Don't try to make disko manage all drives.
- **NFS mounts from dozer** use `x-systemd.automount` + `noauto` + `x-systemd.idle-timeout=5min` pattern.
- If adding a new btrfs host, copy the btrbk config and disko layout from beast or framey — don't invent a different scheme.

### Dotfiles (home-manager)

- Live under `home-manager/dotfiles/<app>/<config>`
- Sway config is in `dotfiles/sway/` (split into `config` + `config.d/exec.conf` + `config.d/idle.conf` + custom `idle-guard.sh`)
- Waybar, mako, kanshi, ghostty configs follow the same pattern
- Ghostty theme is Catppuccin Mocha
- Neovim config is in `dotfiles/nvim/` — Lua-based with lazy.nvim. Do NOT touch it unless asked directly; it's a personal config.

### Git

- Conventional commits: `feat:`, `fix:`, `refactor:`, `revert:`, `chore:`
- `flake.lock` is tracked (NOT in .gitignore) but `home-manager/dotfiles/nvim/undo/` IS ignored
- Keep commit messages factual, not salesy

### Host-specific quirks

| Quirk | framey | beast | pve |
|-------|--------|-------|-----|
| Tailscale interface | `userspace-networking` | `tailscale0` | `tailscale0` |
| Bootloader | lanzaboote (Secure Boot) | systemd-boot | systemd-boot |
| disk encryption | LUKS2 + TPM2 + Secure Boot | none | none (ZFS) |
| Firewall SSH | tailscale0 only | tailscale0 + LAN | LAN physical NIC |
| Home-manager | yes | yes | no |
| `modules/core` | yes | yes | no (inline) |

Copying firewall rules between hosts without adjusting the Tailscale interface name will silently fail.

### pve Samba shares

PVE's Samba shares (`roles/pve/samba.nix`) are intentionally wide-open
(guest-accessible, read/write to the entire dozer-files pool). PVE is an
internal-only server on a trusted LAN — no port exposure to WAN. The
firewall blocks inbound on the physical NIC (`br0`), and Samba only
serves the local subnet. If PVE ever faces an untrusted network, lock
down the shares with `security = user` and per-user credentials.

## Common pitfalls

- **State versions differ**: framey/beast = `23.05`, pve = `24.05`, home-manager = `24.11`. Don't bump them without understanding why they diverged. The `system.stateVersion` comment says "Do not change on a rebuild" — that's not a suggestion.
- **nixpkgs config is set in TWO places**: the flake's `nixConfig` affects `nix build .#` CLI commands. The per-host `nixpkgs.config` affects the system. Both need allowUnfree = true.
- **`allowed-users` / `trusted-users`** are set per-host, NOT in `modules/core/nix.nix`. That's intentional — the core module sets global nix settings, but access control varies.
- **electron insecure packages** (`electron-27.3.11`, `electron-39.8.10`) are permitted on desktop hosts. If you add a new electron app and it fails to build, check whether the electron version needs to be added here.
- **pve does NOT use home-manager** — don't try to import it. It doesn't have `modules/core` either (it has its own minimal nix config inline).
- **beast actually uses Sway-only** despite the comment in flake.nix mentioning Plasma6. The Plasma6 reference is aspirational / historical — ignore it.
- **framey uses `lanzaboote` for Secure Boot** with `systemd-boot.enable = lib.mkForce false`. Don't re-enable systemd-boot on framey.
- **Verify NixOS module options against the pinned nixpkgs source before referencing them in a config patch.** Web docs (NixOS wiki, old PRs, blog posts) lag behind nixpkgs renames by months or years. Real failure: `services.kdeconnect.enable = true` was the path in older nixpkgs; in 25.11 (pinned in flake.lock) the option is `programs.kdeconnect.enable` and the `services.kdeconnect` path was removed entirely — the build fails with `The option 'services.kdeconnect' does not exist`. Before referencing any non-trivial option in a config patch, confirm it exists in `nixpkgs/nixos/modules/.../*.nix` of the flake's pinned input. Quick check: `nix --extra-experimental-features 'nix-command flakes' eval .#nixosConfigurations.<host>.options.<option-path>` against a known-good build — if the option resolves, it's safe to use.
