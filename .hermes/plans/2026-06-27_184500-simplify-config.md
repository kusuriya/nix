# Nix Config Simplification Plan

> **For Hermes:** Use subagent-driven-development to implement this plan task-by-task.

**Goal:** Reduce duplication, fluff, and unnecessary complexity across the ~3-host, ~35-file Nix config. Target: remove ~200 lines of dead/duplicated code while keeping every host exactly as functional.

**Architecture:** Most of the config is already well-modularized. The simplification targets genuine duplication (identical sysctl blocks in two hosts, identical btrbk configs, identical NFS mounts, identical libvirtd setup) plus dead overlay code and stale comments. No restructuring of the module hierarchy — just extracting shared patterns and deleting cruft.

**Tech Stack:** NixOS, Nix Flakes, systemd, btrfs, Sway, libvirt

### What's NOT simplifying

- **PVE standalone** — intentionally avoids `mkSystem` and `modules/core`. This is a server, not a desktop. Simplifying PVE into the shared module pattern would bloat its boot with desktop cruft. Leave as-is.
- **Role modules** — roles are clean and focused. No duplication across them.
- **Home-manager** — 168 lines for dotfiles + programs. Clean enough.
- **Disko files** — host-specific disk layouts, different encryption. No extractable overlap.
- **Kernel module** — 4 lines doing one thing. Not worth changing.
- **Custom packages** — pkgs/ is well-structured.

## Risks, Tradeoffs, and Open Questions

**Q1: Should the shared modules go under `modules/desktop/` or `modules/core/`?** They're desktop-specific (btrfs snapshots, libvirt, sysctl hardening, NFS mounts to a specific NAS). `modules/desktop/` is the right home — `modules/core/` is for things that apply to ALL hosts (including pve, which has ZFS, no btrfs, no sway, no libvirt).

**Q2: The flake.nix `mkSystem` function adds `./hosts/${hostname}` to the module list. Can host-specific modules import shared modules?** Yes — the module system handles transitive imports just fine. Adding `../../modules/desktop/sysctl` in beast's `imports` block will cause Nix to load and merge it. No change needed to flake.nix.

**Q3: Does `mkSystem` also add `modules/core` automatically?** No — each host's `default.nix` explicitly imports `../../modules/core` (and beast/framey do, pve doesn't). The shared desktop modules are layered on top of core.

**Q4: The btrbk instance name becomes `"desktop-snapshots"` — will this conflict if a host is renamed?** No. The systemd unit is `btrbk-desktop-snapshots.timer` which runs hourly. There's no machine name involved. This works on any host that imports the module.

**Risk: OVMF path override.** Both beast and framey currently set OVMF firmware paths via `environment.etc."ovmf/..."`. The extracted module uses `config.virtualisation.libvirtd.qemu.package` to resolve the path. This is fragile — if libvirtd is somehow not enabled in the final configuration, the eval will fail. But libvirtd IS enabled in that same module (and on both current hosts). As long as the module is imported as a unit, it's fine. ⚠️ Consider using `mkIf` to guard:

**Correction for Task 4 — safer OVMF module:**
```nix
{ config, lib, pkgs, ... }:
with lib;
{
  config = mkIf config.virtualisation.libvirtd.enable {
    virtualisation = {
      libvirtd = {
        enable = true;

          package = pkgs.qemu_full;
          runAsRoot = true;
          swtpm.enable = true;
        };
      };
      spiceUSBRedirection.enable = true;
    };
    environment.etc."ovmf/edk2-x86_64-secure-code.fd" = {
      source = "${config.virtualisation.libvirtd.qemu.package}/share/qemu/edk2-x86_64-secure-code.fd";
    };
    environment.etc."ovmf/edk2-i386-vars.fd" = {
      source = "${config.virtualisation.libvirtd.qemu.package}/share/qemu/edk2-i386-vars.fd";
      mode = "0644";
      user = "libvirtd";
    };
  };

```

Hmm, actually the existing code doesn't guard either — it just writes the etc entries assuming libvirtd is enabled. Since these are always used together on these hosts, keeping it simple (no mkIf guard) matches the existing style. The host imports the module and libvirtd comes along. The risk of importing this module without wanting libvirtd is zero in this config.
